 我来为你设计一个C/S架构关系数据库系统的5阶段开发计划。这个计划采用渐进式开发，每个阶段都有可独立运行的成果，同时功能可以串联。

## 技术实现要求
1. 采用c++17编程语言开发，CMake编译
2. 先设计整体的代码目录，再逐步的完成各个阶段的开发
3. 要写设计文档
4. 要写单元测试

## 阶段一：网络通信与协议框架

**核心目标**：建立稳定的C/S通信基础

| 功能模块 | 详细内容 | 验收标准 |
|---------|---------|---------|
| TCP服务器 | 多线程/多进程连接处理，连接池管理 | 支持100+并发连接，稳定运行24小时 |
| 客户端SDK | 基础连接、断开、心跳检测 | 提供C/Python/Java至少一种SDK |
| 通信协议 | 自定义二进制协议（Header+Body），支持SSL/TLS | 协议文档+抓包验证 |
| 命令解析器 | SQL词法分析（Lexer），识别关键字、标识符、常量 | 能正确解析所有DDL/DML关键字 |
| 简单执行器 | 直接返回OK/Error，不实际执行 | 客户端发送任意SQL返回预定义响应 |

**参考实现**：PostgreSQL的Frontend/Backend协议（简化版）、Redis协议

**阶段成果**：`./dbclient -h localhost -p 5432 "SELECT 1"` 能连接并返回模拟结果

## 阶段二：存储引擎与基础DDL

**核心目标**：数据能存能取，支持基础表操作

| 功能模块 | 详细内容 | 验收标准 |
|---------|---------|---------|
| 页式存储 | 8KB/16KB定长页，页头管理，空闲空间管理 | 单表支持100万行，随机读写性能测试 |
| 堆表存储 | 行格式设计（定长字段），TID（页号+槽位）定位 | 插入1万行，逐行读取验证数据完整性 |
| 缓冲区管理 | LRU缓冲池，脏页刷盘，WAL预写日志雏形 | 崩溃恢复测试（kill -9后数据不丢） |
| 系统表 | `pg_class`（表元数据）、`pg_attribute`（列元数据） | 能`SELECT * FROM pg_class`看到所有表 |
| DDL执行 | `CREATE TABLE`（仅定长类型：INT/BIGINT/CHAR/VARCHAR）、`DROP TABLE` | 完整DDL生命周期测试 |
|  catalog缓存 | 表结构缓存，减少系统表查询 | 重复查询性能提升10倍+ |

**参考实现**：SQLite的B-tree、PostgreSQL的Buffer Manager

**阶段成果**：
```sql
CREATE TABLE users (id INT, name VARCHAR(32));
INSERT INTO users VALUES (1, 'Alice');  -- 阶段三才实现，此处仅准备存储
SELECT * FROM pg_class WHERE relname = 'users';  -- 能看到新表
```

## 阶段三：基础DML与事务

**核心目标**：完整CRUD，ACID事务

| 功能模块 | 详细内容 | 验收标准 |
|---------|---------|---------|
| 执行器-v1 | 火山模型（Volcano）迭代器，顺序扫描 | `SELECT * FROM t` 全表扫描正确 |
| 插入/删除 | `INSERT`、`DELETE`（物理删除/标记删除） | 插入后立即可查，删除后不可见 |
| 更新操作 | `UPDATE`（原地更新+迁移更新） | 变长字段更新触发行迁移处理 |
| 表达式计算 | 常量、列引用、算术运算、比较运算 | `SELECT 1+2, id*2 FROM t` 正确计算 |
| 事务管理 | BEGIN/COMMIT/ROLLBACK，MVCC基础（XID，快照） | 并发读写测试，脏读不可见 |
| WAL日志 | 完整Redo日志，崩溃恢复 | 随机kill测试，100%恢复一致性 |
| 锁管理 | 表级共享/排他锁，死锁检测 | 两个会话同时写同一表，一个等待 |

**参考实现**：PostgreSQL的MVCC、SQLite的WAL模式

**阶段成果**：
```sql
BEGIN;
CREATE TABLE accounts (id INT, balance INT);
INSERT INTO accounts VALUES (1, 100), (2, 200);
UPDATE accounts SET balance = balance - 10 WHERE id = 1;
COMMIT;
-- 崩溃后重启，数据必须一致
```

## 阶段四：索引与查询优化

**核心目标**：查询有索引加速，优化器做智能选择

| 功能模块 | 详细内容 | 验收标准 |
|---------|---------|---------|
| B+树索引 | 磁盘B+树，唯一/非唯一，聚簇/非聚簇 | 100万数据，点查<10ms |
| 索引DDL | `CREATE INDEX`、`DROP INDEX` | 在线创建索引不阻塞读写 |
| 索引扫描 | 点查、范围查、Index-Only Scan | `WHERE id = 5`、`WHERE id BETWEEN 1 AND 100` |
| 统计信息 | 表/列基础统计（行数、NDV、直方图雏形） | `ANALYZE` 命令收集统计 |
| 优化器-v1 | 基于代价：选择扫描路径（Seq Scan vs Index Scan） | 小表走全扫，大表点查走索引 |
| 连接操作 | Nested Loop Join，支持等值连接 | 两表JOIN结果正确，内存/磁盘模式 |
| 执行器-v2 | 向量化执行（可选）、物化节点 | TPC-H Q1简化版能跑通 |

**参考实现**：PostgreSQL的B-tree、SQLite的查询规划器

**阶段成果**：
```sql
CREATE TABLE orders (id INT PRIMARY KEY, user_id INT, amount INT);
CREATE INDEX idx_user ON orders(user_id);
-- 优化器选择：user_id = 5 走索引，amount > 0 小表走全扫
EXPLAIN SELECT * FROM orders o JOIN users u ON o.user_id = u.id 
WHERE o.user_id = 5;
```

## 关键技术决策

| 组件 | 推荐方案 | 备选方案 |
|-----|---------|---------|
| 进程模型 | 每连接一进程（PostgreSQL风格） | 每连接一线程 |
| 存储格式 | 堆表+行存储 | 列存储（后期扩展） |
| 索引类型 | B+树（阶段四），后期加Hash/GiST | 一开始就实现多种 |
| 并发控制 | MVCC + 行级锁 | 2PL |
| 日志格式 | 物理Redo + 逻辑Undo | ARIES算法完整版 |
| 网络协议 | 自定义二进制 | 兼容PostgreSQL协议 |

## 开发建议

1. **先跑通再优化**：阶段一先用最简单的方式实现，阶段五再考虑性能极致优化
2. **测试驱动**：每个阶段都要有单元测试、集成测试、故障注入测试
3. **代码规范**：早期定义好错误码体系、内存管理规范、日志格式
4. **参考而非复制**：理解PostgreSQL/SQLite的设计哲学，但代码必须自己写
5. **文档同步**：每个阶段输出架构文档、协议文档、用户手册
